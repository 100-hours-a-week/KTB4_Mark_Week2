# 회고

아쉬운 점 : 클래스 다이어그램 그리고 개발을 시작했는데, 개발하다보니 연관관계를 잘못 생각한 것도 있고, 필요한 메서드가 생각보다 많아서 계속 클래스 다이어그램을 수정하고 개발하고를 반복했습니다. 설계부터 잘해야 고생을 덜하겠구나 느꼈습니다. 

잘한 점 : 클래스 하나 만들때마다 생각을 곰곰히 해보고 만들어본 좋은 경험이었습니다.

## 질문

1. MenuItem의 getRecipeSteps() 메서드에서 recipeSteps를 그대로 반환하고 있는데, 이렇게 되면 외부 클래스에서 반환받은 참조값으로 리스트 내부 값을 수정할 수 있을 것 같습니다. 현재는 외부에서 메뉴 정보나 레시피 순서를 함부로 수정하지 못하게 하기 위해 필드를 protected를 사용했는데, getRecipeSteps()를 통해 결국 수정이 가능한 구조가 되는 것 같아서 고민이 됩니다.이런 경우에는 매번 새로운 리스트를 생성해서 반환하는 방식으로 처리하는 것이 일반적인지 궁금합니다. 또한 new ArrayList<>(recipeSteps) 방식으로 계속 복사해서 반환해도 성능이나 비용 측면에서 큰 문제가 없는지도 궁금합니다.
2. 플레이어를 다중 상속하는 클래스가 생겼을때는 지금처럼 private를 유지해서 값을 다중 생성자로 넘겨야 할까요 아니면 protected로 변경 해서 자식 클래스에서는 참조할 수 있게 바꿔야할까요? 이건 MenuItem을 만들때도 많이 고민했는데 MenuItem인 경우엔 코드가 복잡해진다고 느껴서 protected로 선언했습니다.

## 의도

#### MenuItem 클래스를 추상클래스로 만든 이유

실제 구현하는 클래스가 아니기 때문에 메뉴들의 공통 속성들만 묶어놓는 용도로 추상클래스로 선언했습니다.

#### MenuItem에 필드를 protected로 한 이유

구현화된 자식 클래스에서만 접근 가능하게 설정해서 다른 클래스에서 함부로 못바꾸게 하기 위함 입니다.

먼저 MenuItem 필드를 pirvate로 선언을 해봤는데
생성자를 여러 단계로 전달하는 구조가 코드 흐름을 복잡하게 만든다고 판단해서 protected를 사용하게 되었습니다.

예시) iceAmericano 생성자에서 super로 상위 타입인 drink로 값을 넘기고 drink 생성자에서 super로 다시 MenuItem에 다시 넘겨야 했기때문에 뭔가 과정이 많은 구조라고 생각해서 사용하지 않았습니다.

#### MenuItem에 executeStep(실제 제조 메소드)을 추상메서드로 한 이유

각 메뉴마다 제조 과정이 다르기 떄문에 각 구현 클래스에서 오버라이딩해서 사용하고 싶어서 추상클래스로 했습니다.

#### MenuItem에 메서드를 public으로 한 이유

레시피 정보 출력, 레시피 정답 체크, 이름 조회, 가격 조회는 외부 클래스에서 사용하는 메서드기 때문에 public으로 했습니다.

#### Dessert 클래스의 addCream() 메서드와 decorator() 메서드를 추상클래스로 만든 이유

다른 dessert 메서드는 현재 프로젝트에 있는 모든 디저트가 공유하는 동작이지만, addCream과 decorator는 케이크별로 다르다고 생각해서 추상클래스로 만들고 구현 클래스에서 오버라이딩으로 기능 각각 다르게 완성하게 하려고 만들었습니다.

#### 열거형을 사용한 이유

메뉴 생성 및 제조 과정에서 사용되는 값들은 정해진 상수 집합이라고 판단하여 열거형을 사용했습니다.

또한 열거형은 JVM에 하나씩만 올라가기 때문에 == 비교가 가능하기 때문에 equals를 쓰지 않아서 NPE에서 안전하다고 느꼈습니다.

#### interface(AddGreenTeaPowder, ExtractShot)의 의미

Drink 클래스는 모든 음료에 대한 공통 클래시지만 메뉴마다 디테일에서 차이가 있는 부분을 처리하기 위해 두개의 인터페이스를 추가해서 각각 구현클래스에서 오버라이딩해서 동작을 수행했습니다.
